## Context

当前系统具备：
- Next.js 16 前端与 API（含 SSE 流式输出）
- Python Agent runtime（JSON Lines 协议）
- 现有 `pharma_agent` 与 `employee_investment_team` 能力

本次需要新增一个“可直接产出投资建议”的 Agent 能力，并在现有聊天/工作台链路中可用。关键约束：
- 需复用现有 Agent 注册、执行与流式协议，不引入破坏性协议变更。
- 输出必须包含风险提示与能力边界，避免将系统行为表述为确定性投资承诺。
- 第一阶段优先交付可落地的建议流程，不强依赖新增外部行情系统。

## Goals / Non-Goals

**Goals:**
- 提供一个金融投资建议 Agent，能接收结构化输入并输出结构化建议。
- 建立建议生成最小闭环：目标识别 -> 风险评估 -> 资产配置建议 -> 行动清单 -> 风险提示。
- 在现有 Dashboard 与 API 中无缝接入，支持流式输出与会话留存。
- 引入安全护栏：免责声明、信息不足时澄清、超边界降级。
- 提供可插拔工具层，覆盖文件读取、搜索、股票趋势、skills 加载，并预留风险评估与组合建议扩展。
- 建立 planner、executor、evaluator 多 Agent 协作链路，支持多轮评估与补充执行后再输出结果。

**Non-Goals:**
- 不在本阶段实现真实交易下单、自动调仓执行。
- 不在本阶段实现高频实时行情引擎或复杂量化回测平台。
- 不承诺个股/基金收益结果，不提供确定性收益保证。

## Decisions

### Decision 1: 采用“planner + executor + evaluator”多 Agent 协作方案
- Choice:
  - planner 负责将用户问题拆解为执行计划（含工具步骤、输入输出契约、优先级）。
  - executor 负责按计划调用工具与 skills，汇总证据并生成候选答案。
  - evaluator 负责验证候选答案的正确性、覆盖度与需求匹配度，并在不满足阈值时触发补充计划。
  - 编排器控制多轮循环上限与终止条件，最终输出审阅后的结果。
- Rationale:
  - 三角色分工可以显著降低“直接回答但证据不足”的风险。
  - evaluator 能将预算约束、用户显式需求与信息完整性纳入统一验收标准。
- Alternatives considered:
  - 单 Agent 直接调用工具：实现简单，但缺少独立审核环节，难以保障信息全面性。
  - 仅 planner+executor 双角色：评估责任不清晰，无法稳定实现质量闸门。

### Decision 2: 输出采用“结构化建议对象 + 可读文本”双轨
- Choice:
  - Agent 结果包含稳定结构字段（如 risk_profile、allocation_plan、actions、disclaimer）和人类可读总结。
- Rationale:
  - 前端可稳定渲染卡片与后续自动化处理，同时保留对话可读性。
- Alternatives considered:
  - 仅返回自由文本：开发快，但无法可靠做前端结构化展示和审计。

### Decision 3: 安全护栏前置到生成流程
- Choice:
  - 每次建议前检查输入完整性与能力边界，触发澄清或降级路径。
- Rationale:
  - 在生成前控制风险比生成后补救更稳定，且便于测试覆盖。
- Alternatives considered:
  - 仅在输出末尾附免责声明：无法阻止错误前提下的建议内容生成。

### Decision 4: 数据源策略采用“阶段化增强”
- Choice:
  - Phase 1 使用用户输入 + 规则化基线，保留数据适配器接口。
  - Phase 2 可接入行情/研究数据源。
- Rationale:
  - 先交付业务闭环，再逐步提升数据深度，减少首版依赖风险。

### Decision 5: 工具体系采用“P0/P1/P2 分层交付”
- Choice:
  - P0（首发必需）: 文件读取（文本/PDF/图片）、搜索、股票趋势、skills 加载、合规护栏执行。
  - P1（建议紧随）: 风险评估、组合建议。
  - P2（上线前增强）: 可观测与审计（工具调用轨迹、输入摘要、模型版本）。
- Rationale:
  - 与产品风险优先级一致，先保证可用与安全，再增强建议质量与审计能力。
- Alternatives considered:
  - 一次性全量交付：实现复杂度高且联调成本大，首版不利于快速验证。

## Risks / Trade-offs

- [风险偏好识别不准导致建议偏差] -> 通过必填澄清问题与默认保守策略降低风险。
- [用户将建议视为个性化投顾结论] -> 在响应首段和尾段均展示免责声明，并标注假设前提。
- [首版不接行情导致建议时效性有限] -> 输出中显式声明数据时效边界，并设计后续可插拔行情接口。
- [结构化输出字段后续演进带来兼容问题] -> 版本化结果 schema，前端按版本解析。
- [工具调用失败导致建议中断] -> 对每类工具定义降级策略（重试、回退到保守建议、清晰错误提示）。
- [外部行情数据延迟或波动] -> 趋势工具输出时附带时间戳与数据来源，超过阈值提示时效风险。
- [多轮协作导致时延上升] -> 设置循环轮次上限与快速终止规则，并向前端回传阶段进度。
- [planner 计划粒度不稳定] -> 定义计划 schema（步骤类型、必需输入、成功判定）并做静态校验。

## Migration Plan

1. 新增 planner、executor、evaluator 角色模块与编排器，不改动现有 Agent 通信协议。
2. 定义计划 schema、执行结果 schema、评估报告 schema，并串联循环控制逻辑。
3. 在 Agent 注册层加入新协作系统入口，确保 Dashboard 可发现。
4. 扩展前端输入与展示（风险偏好、目标、期限、预算等字段），并展示阶段化执行反馈。
5. 增加 API/Agent 交互参数映射与结构化结果渲染。
6. 增加测试：协议兼容、核心建议路径、澄清路径、免责声明必出、评估回路正确触发。
7. 工具分层交付：先上线 P0 工具，再补齐 P1/P2。
8. 灰度发布：先内部环境验证，再逐步开放。

Rollback strategy:
- 通过配置下线新 Agent（从 registry 隐藏或禁用），不影响既有 Agent。
- 回滚仅涉及新增模块，保持数据库与协议兼容。
