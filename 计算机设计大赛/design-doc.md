## Problem Statement

为计算机设计大赛构建一个"智能体赋能的医药生物投资决策支持系统"。当前市场上，同花顺有通用 Agent 平台但缺乏医药领域深度，Wind 有权威数据但 AI Agent 能力弱，药融云（摩熵医药）有医药专业数据但没有投资决策 Agent。**没有产品占据"医药领域专业 Agent + 投资决策闭环"这个交叉点。**

## What Makes This Cool

三大功能融合形成完整的医药投资决策闭环：
1. **副驾模式（Copilot）** — 用户描述想了解的药企/赛道，多个专业 Agent 自动编排分析任务，生成带数据溯源的深度报告
2. **Multi-Agent 协作团队** — 财报分析师、研报解读员、政策监控员、竞品追踪员协同工作，像一个完整的分析师团队
3. **风险雷达 + 推演引擎** — 实时监控药企风险信号并主动预警，更独特的是能**模拟政策变化、竞品获批等场景对药企的影响**（"医药投资沙盘"）
4. **员工画像 Skill + 投资小组协作** — 系统持续记录每位员工的投资行为、观察记录、分析习惯和常用 SOP，将其沉淀为可调用的“员工 Skill”。员工发起投资决策时，可快速拉起一个由不同专业方向组成的小组，自动汇聚数据分析、财报分析、政策监控等多角色观点，并将关键建议同步推送到该员工的社交平台账号

## Constraints

- **团队规模：** 3人团队，使用 vibe coding 模式（AI 辅助开发），开发效率高
- **时间限制：** 国赛决赛 2026年7月17日-21日（约4个月开发时间）
- **数据来源：** 官方"数睿思"平台提供财报、研报、宏观数据；可自增药监局审评审批数据、临床试验注册信息
- **技术栈：** LangChain/LangGraph 自建 Agent，Python 后端
- **LLM 推理：** 使用商业 API（GPT-4/Claude），无需离线演示
- **提交物：** PDF设计报告（≤30页）、源码+部署包、5-8分钟演示视频
- **评审加分项：** 多模态交互、自主学习迭代、低资源场景适配、高并发处理

## Premises

1. 官方样本数据足够支撑核心功能演示
2. LangChain/LangGraph 技术栈能支撑 Multi-Agent 架构
3. 三大功能全做，每个功能聚焦 1-2 个核心场景深度打磨。场景1-3为P0必须完成，场景4-5为P1拉伸目标
4. 技术栈统一使用 LangChain/LangGraph，不引入 Dify 避免技术栈分裂

## Approaches Considered

### Approach A: "精准打击" — 最小可行演示
聚焦 3 个核心场景的完整 demo，用 LangGraph 构建 3 个专业 Agent。开发量可控（2-3周），风险低，但功能数量少可能不够震撼。

### Approach B: "全家桶" — 三大功能全做
副驾模式 + Multi-Agent 协作 + 风险雷达全部实现。覆盖赛题四大维度，展示面广，但开发量大（4-6周），需要严格时间管理。

### Approach C: "创意破局" — 投资决策推演引擎
独特切入点：模拟政策变化、竞品获批等场景对药企估值的影响。创新性极高但实现难度大。

## Recommended Approach

**B+C 混合方案** — 三大功能全做（Approach B），同时在风险雷达模块中融入场景推演能力（Approach C 的精华）。

### 赛题四大维度映射

| 赛题维度 | 系统功能 | 核心场景 |
|---------|---------|---------|
| 智能问数服务 | Copilot 多轮对话 + 财报/研报 Agent | "恒瑞医药近3年研发投入占比及管线进展？" |
| 运营水平评估 | 多维度诊断 Agent | 销售费用率、研发强度、临床成功率综合评分 |
| 风险与机会洞察 | 风险雷达 + 推演引擎 | 模拟集采政策对某药企收入影响、竞品获批对市场格局影响 |
| 定制内容生成 | 报告生成 Agent（带溯源） | 个股深度分析报告，每条结论标注来源页码 |
| 团队智能协作 | 员工画像 Skill + 投资小组协同 | 某员工一键拉起“数据分析 + 财报分析 + 政策监控”讨论组，形成多视角投资建议并同步通知 |

### 技术加分项对应

| 加分项 | 实现方式 |
|-------|---------|
| 多模态交互 | 支持图表交互展示、PDF 上传解析、可视化数据探索 |
| 自主学习迭代 | Agent 根据用户反馈调整分析策略（偏好检索 + prompt 注入） |

| 高并发处理 | 异步 Agent 调度、结果缓存、流式输出 |
| 团队经验沉淀 | 将员工投资行为、观察记录与 SOP 自动抽取为可复用 Skill，支持组织级经验复用 |

### 核心演示场景（Demo 重点）

1. **场景1：个股深度分析** — 输入"分析恒瑞医药的投资价值"→ 4个Agent并行工作 → 生成带溯源的综合报告
2. **场景2：风险预警推演** — 展示某药企面临集采政策风险 → 推演引擎模拟影响 → 生成应对建议
3. **场景3：多轮对话追问** — 基于场景1的报告继续追问"它和复星医药比谁更好？" → Agent 自动组织对比分析
4. **场景4：运营健康度评估** — 输入一组药企 → 自动生成多维度评分雷达图
5. **场景5：自定义报告生成** — 选择模板+药企 → 生成 PDF 报告，每条结论可点击溯源到原文
6. **场景6：员工投资小组快速拉起** — 某员工发起“是否投资某创新药企”的决策请求 → 系统按其职位标签与画像自动组建“数据分析 + 财报分析 + 政策监控 + 风险控制”小组 → 输出多角色讨论纪要、综合建议与分歧点
7. **场景7：社交通知与经验回流** — 小组形成阶段性建议后，系统将摘要、风险提示和待确认事项推送到相关员工社交平台，并将员工反馈继续写回画像与 SOP 库

## 推演引擎技术规格

推演引擎是核心差异化功能，采用 **"参数化场景模板 + LLM 推理"** 混合方法：

### 方法论
- **不是**纯 LLM 幻想式预测，而是基于历史数据的参数化推演
- 预定义 3-5 个场景模板（如"集采纳入"、"竞品获批"、"临床失败"）
- 每个模板有明确的输入参数、影响因子和输出维度

### 工作流程
1. 用户选择场景类型（如"集采政策影响"）
2. 系统从 RAG 中检索该药企的历史数据（营收构成、产品线、可比公司案例）
3. 检索 **已发生的同类事件** 作为 ground truth（如"XX药品2023年纳入集采后营收变化"）
4. LLM 基于历史案例 + 当前药企数据，推理影响幅度并给出置信区间
5. 输出：影响摘要 + 关键指标变化估算 + 历史案例参考 + 不确定性说明

### 验证策略
- 选择 1-2 个已知结果的历史事件（如"恒瑞医药2021年集采影响"）作为验证基准
- 系统给出的推演结果与实际结果对比，展示推演可信度

### 示例输入输出
```
输入：恒瑞医药 + 集采政策（假设核心产品纳入第X批集采）
输出：
  - 预计营收影响：-8%~-15%（基于同类药企历史案例）
  - 参考案例：XX药业2022年集采后营收同比下降12%
  - 缓冲因素：创新药管线贡献占比逐年提升
  - 置信度：中（基于3个可比案例）
```

## 数据模型

### 财报向量库
- 分块策略：按章节分块（经营分析、财务报表、管理层讨论），每块 512-1024 tokens
- 元数据：公司代码、报告年份、章节名、页码（用于溯源）

### 研报向量库
- 分块策略：按段落分块，保留标题层级
- 元数据：券商、分析师、发布日期、评级、目标价、页码

### 宏观数据库
- 结构化存储（SQLite/PostgreSQL）：医疗CPI、医保支出、行业政策时间线
- 时间序列格式，支持趋势查询

### 药审数据库（自增）
- 结构化存储：药品名称、申请类型、审评状态、审批日期、适应症
- 来源：CDE 药审中心公开数据

### 员工画像与 Skill 库（新增）
- 员工基础档案：员工ID、姓名、岗位、部门、职位标签、覆盖专业方向
- 投资行为记录：关注标的、研究轨迹、历史建议、最终决策、收益/偏差复盘
- 观察记录表：日常市场观察、政策观察、竞品事件摘录、个人判断备注
- SOP 抽取结果：信息收集步骤、指标关注顺序、常用判断框架、风险排查清单
- Skill 表达形式：将员工画像、职位标签、专业边界、典型决策风格封装为可调用的员工 Skill，供其他员工在决策时引用
- 通知映射：员工ID 与社交平台账号绑定，用于推送系统建议、讨论摘要和风险预警

## 员工画像 Skill 与投资小组机制

### 目标
- 不把员工经验停留在个人脑中，而是沉淀为组织可复用的“数字化投资技能”
- 支持员工在投资决策时快速拉起跨专业小组，而不是依赖临时沟通和人工组织
- 将讨论结果同步到员工外部沟通渠道，形成“系统建议 - 员工反馈 - 画像更新”的闭环

### 核心机制
1. 系统持续采集每位员工的投资行为、观察记录、分析路径和复盘结果
2. 通过规则抽取 + LLM 归纳，生成员工画像、岗位标签和个性化 SOP
3. 将该员工封装为一个可调用的“员工 Skill”，代表其专业视角与决策偏好
4. 当任一员工发起投资决策时，Copilot 编排层根据任务类型自动组建一个“小组”
5. 小组默认由不同专业方向的 Skill/Agent 组成，如数据分析、财报分析、政策监控、风险控制
6. 系统输出统一的讨论纪要，包括共识、分歧、建议动作、待验证事项
7. 关键决策建议、预警或待办通过社交平台通知相关员工，并将员工反馈写回画像库

### 小组编排策略
- 按员工职位标签确定默认参与角色，例如研究员优先拉起财报分析与数据分析，合规/风控岗位优先拉起政策监控与风险控制
- 按标的类型和场景动态扩展角色，例如创新药项目增加临床与竞品跟踪，仿制药项目增加集采与政策推演
- 同时保留“员工本人画像 Skill”参与讨论，确保建议既反映团队共识，也保留个人风格

### 输出与通知
- 讨论输出：多角色摘要、支持证据、反对证据、最终建议、风险提示、需要人工确认的事项
- 通知输出：将关键建议摘要、风险变化、待确认结论同步到员工绑定的社交平台账号
- 回流机制：员工在社交平台或系统内的反馈会被回写到画像、SOP 和 Skill 更新流程中

## Agent 错误处理与可观测性

### 输出验证
- 每个 Agent 返回结果时附带 confidence_score（0-1）
- confidence < 0.6 时标记为"低置信度"，前端高亮提示用户
- 关键数据点（营收、利润等）与原始数据交叉验证

### 容错策略
- 单个 Agent 超时（30s）或异常时，Copilot 编排层跳过该 Agent 并标注"该维度分析暂不可用"
- 不因单 Agent 失败阻塞整体报告生成

### 可观测性
- LangSmith 追踪每次 Agent 调用的输入/输出/耗时/token消耗
- 结构化日志记录每步决策链路，用于调试和演示"思考过程"

## 自学习机制

- 用户对分析结果点赞/点踩，存入偏好数据库
- 下次分析同类问题时，从偏好数据库检索用户历史反馈
- 将历史反馈注入 Agent 的 system prompt 中作为 few-shot 引导
- **不做模型微调** — 竞赛环境下 prompt 工程 + 偏好检索已足够

## What I noticed about how you think

- 当我给出三个产品方向选项时，你说"我希望这三个都包含"。你不接受妥协，想要完整的产品形态。这种野心在竞赛中是优势——评审喜欢看到完整的系统思考。
- 你选择了 LangChain 自建而非 Dify 低代码路线。这说明你对技术实现有掌控力的要求，不满足于拖拽式开发。
- 最终选择 B+C 混合方案——全部功能加上创新性推演引擎。你既要覆盖面，也要差异化亮点。这是"贪心但有策略"的思维方式。
